PRODUCT REQUIREMENTS — DNSSwitcher Apple Silicon Migration + Security Hardening
Version: 2.0.0
Last Updated: 2026-06-11
==============================================================


1. PURPOSE AND STRATEGIC CONTEXT
---------------------------------
DNSSwitcher is a macOS menu bar utility (4 Swift source files) that lets users
quickly switch DNS settings across named profiles. Originally created by Matt
McNeeney in 2016 (github.com/mattmcneeney/DNSSwitcher) using Xcode 7.3.1, Swift
2.x, and CocoaPods 0.39.0. The original binary is Intel-only (x86_64).

This fork pursues two goals:

  Phase 1 (complete): Produce a notarized, code-signed universal binary
  (arm64 + x86_64) that runs natively on Apple Silicon Macs with no Rosetta
  layer, passes Gatekeeper, and can be distributed outside the App Store.

  Security hardening (complete): A deepsec scan identified one HIGH-severity
  RCE, one MEDIUM path-traversal, and two stability bugs. All four were
  confirmed true-positives and fixed in this fork.

Target user: the original developer and any Mac users on Apple Silicon hardware.


2. USER STORIES
---------------
US-01  As an Apple Silicon Mac user, I want DNSSwitcher to run natively so that
       I do not see macOS compatibility warnings and the app does not stop launching.

US-02  As a developer distributing DNSSwitcher, I want the binary to be notarized
       so that Gatekeeper allows users to open it without security prompts.

US-03  As a developer, I want a repeatable release pipeline so that future versions
       can be built, signed, notarized, and distributed without manual guesswork.

US-04  As a user, I want the app to be free of exploitable security vulnerabilities
       so that a malicious config file or symlink cannot execute arbitrary code on
       my machine.


3. FUNCTIONAL REQUIREMENTS
---------------------------
3.1 Dependency Modernisation (Phase 1)
  - CocoaPods tooling updated to current stable version (1.x+).
  - SwiftyJSON dependency updated from commit-pinned 2.3.2 to semver ~> 5.0.
  - pod install completes without errors.
  - SwiftyJSON.framework built for arm64 (verified with lipo).

3.2 Swift Language Migration (Phase 1)
  - All four source files compile cleanly in modern Xcode (16.x) with Swift 5/6.
  - No use of removed Swift 2 APIs (NSTask, NSPipe, NSBundle, NSFileManager,
    NSData, NSDate, NSAlertStyle, componentsSeparatedByString, etc.).
  - @objc inference on selector-referenced methods restored explicitly.
  - NSMenuDelegate and NSApplicationDelegate method signatures updated to Swift 5
    conventions (external label _ convention).

3.3 Universal Binary Build (Phase 1)
  - ARCHS = $(ARCHS_STANDARD) set in both Debug and Release configurations.
  - MACOSX_DEPLOYMENT_TARGET raised from 10.11 to 12.0 (Monterey).
  - Successful xcodebuild on both arm64 and x86_64.
  - lipo -info confirms fat binary containing both architectures.

3.4 Security Hardening (Phase 1 — complete)
  - load_cmd feature removed from SettingItem and AppDelegate (HIGH — RCE via
    ~/.dnsswitcher.json). The load_cmd JSON field and its Process() execution path
    are deleted; no substitute is provided.
  - Symlink guard added to createDefaultConfigFile (MEDIUM — path traversal). Before
    writing, the destination is resolved via resolvingSymlinksInPath() and compared
    to the expected path; a mismatch aborts the write.
  - Bounds guard added to highlightEnabledInterface failover path (BUG — crash on
    empty interface list). guard !self.interfaceMenu.items.isEmpty added before
    items[0] subscript.
  - stderr redirected to the same Pipe as stdout in runCommand (BUG — blank error
    dialogs). task.standardError = pipe added alongside task.standardOutput = pipe.

3.5 Notarization & Distribution (Phase 2)
  - Developer ID Application certificate provisioned and installed.
  - Code signing configured in Xcode project (DEVELOPMENT_TEAM, CODE_SIGN_IDENTITY).
  - xcrun notarytool credentials stored in keychain.
  - Full pipeline tested: archive → export → notarize → staple → Gatekeeper pass.
  - Release process documented in RELEASING.md.


4. NON-FUNCTIONAL REQUIREMENTS
--------------------------------
4.1  The app must launch without Rosetta on Apple Silicon hardware.
4.2  The notarized binary must pass `spctl --assess --verbose` without errors.
4.3  The release pipeline must be executable by a single developer without
     interactive Xcode GUI steps (command-line reproducible).
4.4  No new third-party dependencies may be introduced beyond updating SwiftyJSON.
4.5  The deployment target (macOS 12) must not break any existing functionality
     tested on macOS 12, 13, 14, and 15.

9. OUT OF SCOPE
----------------
- App Store distribution (requires different signing, entitlements, sandboxing).
- Adding new DNS features or UI changes.
- Re-introducing a load_cmd equivalent (removed as an exploitable RCE vector;
  any future pre-flight hook capability requires a full security design review).
- Migrating from CocoaPods to Swift Package Manager (deferred — SPM migration
  is a separate initiative that would remove the Pods/ directory entirely).
- Updating the bundle identifier or version number.
- CI/CD automation (GitHub Actions, Fastlane) — manual pipeline is sufficient
  for this single-developer project.


10. OWNER DECISIONS (RESOLVED)
--------------------------------
Q1: Raise deployment target to macOS 12 or higher?
    DECISION: macOS 12 (Monterey). Drops support for Big Sur (11) and older, which
    is acceptable given the app's user base and Apple Silicon hardware requirements.

Q2: Universal binary or arm64-only?
    DECISION: Universal binary (arm64 + x86_64). Ensures the app continues to work
    for users on Intel Macs who may not have upgraded hardware yet.

Q3: Keep CocoaPods or migrate to Swift Package Manager?
    DECISION: Keep CocoaPods for this migration. SPM migration is deferred to avoid
    scope creep — the goal is Apple Silicon support, not build system modernisation.
